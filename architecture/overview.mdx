## Design philosophy

APM Enterprise is built on a strict **Separation of Concerns (SoC)** pattern. Every file in the codebase belongs to exactly one of three layers вЂ” Presentation, Business Logic, or Data/Infrastructure вЂ” and is never allowed to reach across two layers at once.

This was chosen for three concrete reasons:

1. **Testability** вЂ” each layer can be exercised in isolation. Hooks never import React components; service functions never import hooks.
2. **Replaceability** вЂ” the mock server (`mockServer.js`) intercepts `window.fetch` at the infrastructure layer. Replacing it with a real backend requires no changes above the `services/` directory.
3. **Predictable data flow** вЂ” data moves in one direction: Infrastructure в†’ Business Logic в†’ Presentation. The only upward signal is a global `CustomEvent` used for error broadcasting (see [Data Flow](/architecture/data-flow)).

## System layers

The application is divided into three distinct layers.

| Layer | Responsibility | Key files |
| --- | --- | --- |
| **Presentation (UI)** | Render data, capture user input, delegate to hooks | `pages/`, `components/`, `App.jsx` |
| **Business Logic** | Async state, form validation, derived data via `useMemo` | `hooks/` |
| **Data / Infrastructure** | HTTP transport, API contracts, fake database | `services/` |

Pages receive everything they need from a hook and never call `fetch` or a service function directly. Hooks compose other hooks and call service functions but never import components. Service functions call `httpClient` and nothing else.

**`pages/`** вЂ” One file per application route. Each page component imports a single domain hook and a set of atomic components. It contains only JSX and wiring; no business logic.

**`components/`** вЂ” Stateless or minimally stateful UI primitives. They accept props and render markup. They have no knowledge of API calls, routing, or application state.

**`hooks/`** вЂ” Every piece of business logic: data fetching, form handling, filtering, authentication state. Hooks compose other hooks and call service functions.

**`services/`** вЂ” Three distinct concerns in one directory: `httpClient.js` is the low-level transport, `api.js` defines the APM Enterprise API contract, `apiClient.js` calls JSONPlaceholder, and `mockServer.js` intercepts all network traffic for local development.

**`utils/`** вЂ” Pure functions with no side effects. `cn()` merges Tailwind class strings safely. `money()` formats numbers as currency strings.

## Navigation pattern

APM Enterprise uses **tab-based SPA routing** driven by an `activeTab` string in `App.jsx` вЂ” not React Router or any browser history API.

```jsx
// src/App.jsx
const [activeTab, setActiveTab] = useState('dashboard');

// ...

<Layout user={user} activeTab={activeTab} setTab={setActiveTab}>
    {activeTab === 'dashboard' && <DashboardPage />}
    {activeTab === 'projects' && <ProjectsPage />}
    {activeTab === 'users' && <UsersPage onError={handleError} />}
    {activeTab === 'config' && ( /* placeholder */ )}
</Layout>
```

`Layout.jsx` renders the sidebar and passes `setTab` down to each `NavItem` button. Clicking a nav item calls `setTab` with the new tab name, which triggers a React re-render that conditionally mounts the correct page.

This approach was chosen because the application is a single-screen dashboard. There are no deep-linkable URLs, no back/forward navigation requirements, and no need for code-splitting by route. Adding React Router would introduce history management complexity with no user-visible benefit for this use case.

<Note>
The `config` tab renders a placeholder вЂ” the configuration module is not yet implemented.
</Note>
