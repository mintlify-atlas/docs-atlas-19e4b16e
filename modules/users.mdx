## Overview

The Users module fetches a live directory of 10 users from the [JSONPlaceholder](https://jsonplaceholder.typicode.com) public API. It demonstrates external API integration within the APM Enterprise platform — data is not sourced from the internal mock server but from a real HTTP endpoint.

The module is built on the `useUsers` hook, which calls `apiClient.getUsers()`. Any errors are surfaced upward through an `onError` prop to the application-level `ErrorModal`.

---

## User table

Users are displayed in a full-width table with four columns:

| Column | Content |
|---|---|
| **Identidad** | User's full name (bold) and email address (muted). A colored avatar tile showing the first letter of the name is displayed on the left. |
| **Organización** | Company name (bold) and company business tagline (`company.bs`, muted). |
| **Dominio** | External link to `https://{user.website}`, opens in a new tab. |
| **Auditoría** | **Inspeccionar** button — navigates to the detail view and triggers deep audit logging. |

While data is loading, the table body shows 5 animated skeleton rows.

## Pagination

The user table is paginated at 5 users per page. The footer of the table card shows **Página X de Y** and provides two navigation buttons:

- **ChevronLeft** — previous page, disabled on page 1
- **ChevronRight** — next page, disabled on the last page

```js
// src/pages/UsersPage.jsx
const filtered = users;
const totalPages = Math.ceil(filtered.length / pageSize);
const paginated = filtered.slice((page - 1) * pageSize, page * pageSize);
```

With 10 users and `pageSize = 5`, the table renders across 2 pages.

## Deep audit traceability

When **Inspeccionar** is clicked, in addition to navigating to the detail view, the button handler fires two sequential API calls to JSONPlaceholder and logs structured results to the browser console using `console.group`.

```jsx
// src/pages/UsersPage.jsx
onClick={async () => {
    setSelectedUser(u);
    try {
        console.group(`🔍 Auditoría Técnica: Usuario [${u.id}] ${u.name}`);
        console.log("🚀 Iniciando trazabilidad de red profunda...");

        const posts = await apiClient.getUserPosts(u.id);
        console.log(`📦 Posts detectados (${posts.length}):`);
        console.table(posts.slice(0, 3));

        if (posts.length > 0) {
            const comments = await apiClient.getPostComments(posts[0].id);
            console.log(`💬 Comentarios en el post insignia [${posts[0].id}]:`);
            console.table(comments.slice(0, 3));
        }

        console.log("✅ Trazabilidad completada con éxito.");
        console.groupEnd();
    } catch (err) {
        console.warn("⚠️ Auditoría profunda limitada:", err.message);
        console.groupEnd();
    }
}}
```

The audit trace follows this sequence:

1. Open a named `console.group` labeled with the user's ID and name.
2. Call `apiClient.getUserPosts(userId)` — fetches all posts by the user from `GET /posts?userId={id}`.
3. Log the first 3 posts as a `console.table`.
4. If posts exist, call `apiClient.getPostComments(posts[0].id)` — fetches comments on the user's first post from `GET /comments?postId={id}`.
5. Log the first 3 comments as a `console.table`.
6. Close the group with `console.groupEnd()`.

If either API call throws, the `catch` block logs a warning and still closes the group cleanly.

<Tip>
To see the deep audit logs, open your browser **DevTools** (`F12` or `Cmd+Option+I`), navigate to the **Console** tab, and click **Inspeccionar** on any user. Look for the collapsible group labeled `🔍 Auditoría Técnica: Usuario [N] Name`. Expand it to see the `console.table` output for posts and comments. Enable **Verbose** log level in the Console filter if you do not see all entries.
</Tip>

## Error propagation

The `useUsers` hook normalizes all fetch errors into a `{ status, message }` object:

```js
// src/hooks/useUsers.js
} catch (err) {
    setUsers([]);
    const msg = err.status === 404 ? "Usuario no encontrado" :
        err.status === 401 ? "Sesión no autorizada" :
            (err.message || "Falla de auditoría de red");
    setError({
        status: err.status || 500,
        message: msg
    });
}
```

`UsersPage` receives an `onError` prop from its parent. When `error` is set, the component calls `onError(error.status, error.message)`, which triggers the application-level `ErrorModal`:

```jsx
// src/pages/UsersPage.jsx
useEffect(() => {
    if (error && onError) {
        onError(error.status || 500, error.message || "Falla de Auditoría");
    }
}, [error, onError]);
```

Unlike the Projects module (which renders its own inline error banner), the Users module delegates error display entirely to the parent via `onError`. This keeps error UI consistent at the application shell level.

<Note>
The `useUsers` hook also subscribes to the `apm:api_status` window event. Calling `window.setSimulatedError(code)` from the browser console will clear the user list and re-trigger `fetchUsers`, which will fail and propagate the error through `onError` to the `ErrorModal`.
</Note>
